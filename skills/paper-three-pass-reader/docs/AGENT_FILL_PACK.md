# Agent Fill Pack — paper-three-pass-reader (v0.2.1-alpha)

> New in v0.2.1-alpha. Designed to make the runner output directly
> usable by another agent (or a human) for follow-up reading work.

The Agent Fill Pack is a directory generated by `run_paper_reading.py --fill-pack`. It contains step-by-step task files that guide an agent or human through turning a fresh runner draft into a real `paper_reading.json`.

## When to use

Use `--fill-pack` whenever you:

- Want to hand a draft off to another agent to fill in.
- Need a checklist of which fields are present / draft / missing / unavailable.
- Plan to upgrade a weak-mode draft (`abstract_only` / `screenshot_only` / `partial_text`) to `full_text` after acquiring the body.

## Generated layout

For a run at `runs/<root>/<slug>/`, the runner writes `runs/<root>/<slug>/fill-pack/`:

```
fill-pack/
├── 00_README.md                  # Overview, what's doable, what's not, re-render / re-publish commands
├── 01_stage0_intake_resolution.md
├── 02_pass1_five_cs.md
├── 03_pass2_main_ideas.md
├── 04_claims_evidence_map.md
├── 05_figures_tables.md
├── 06_pass3_reconstruction.md
├── 07_critical_review.md
├── 08_reproduction_plan.md
├── 09_finalize_json.md
├── 10_quality_gate.md
├── prompts.json                  # Machine-readable prompts per stage (zh-CN or en)
├── field_checklist.json          # Per-field status: present / draft / missing / unavailable_due_to_reading_mode / needs_verification
└── draft_status.json             # Aggregate status + recommended_next_action
```

## Status taxonomy

| Status | Meaning |
| --- | --- |
| `present` | Field is filled with real content. |
| `draft` | Field still contains `[DRAFT]` placeholder. |
| `missing` | Field is `null` or empty. |
| `unavailable_due_to_reading_mode` | Field legitimately cannot be filled given the current reading mode (only set in `abstract_only` / `screenshot_only` / `partial_text` for Pass 2 / Pass 3 content). |
| `needs_verification` | Boolean derived flag — true if `status` is `draft` or `unavailable_due_to_reading_mode`. |

## Reading-mode discipline

The fill pack adapts its instructions to the input kind and reading mode:

- **full_text** — All steps are fillable. `correctness` / `clarity` should carry `[Paper evidence]` or `[Agent inference]`.
- **partial_text** — Pass 1 (Five Cs) is fillable from abstract + metadata. Pass 2 / Pass 3 partially fillable.
- **abstract_only** — Pass 1 skeleton only. Pass 2 / Pass 3 explicitly marked `unavailable_due_to_reading_mode`.
- **screenshot_only** — Same as `abstract_only`, plus the runner requires `intake_quality.missing_fields` to mention `full body` / `references` / `figures` / `tables`.

The `04_claims_evidence_map.md` step is the strictest. In weak modes the only allowed labels are `[Author claim]`, `[Uncertain]`, `[Needs verification]`. `[Paper evidence]` and `[Figure/Table evidence]` are NOT available because no body was read.

## Agent profile

`--agent-profile` (default / strict / beginner / researcher / engineer) tunes the tone of the fill-pack instructions.

| Profile | Tone |
| --- | --- |
| `default` | Balanced, generic reader. |
| `strict` | Insists on evidence labels, no over-claims, downgrades weak claims. |
| `beginner` | Explains each step more slowly, calls out common mistakes. |
| `researcher` | Emphasizes novelty / context / related work. |
| `engineer` | Emphasizes reproducibility / implementation details. |

> In v0.2.1-alpha, the profile is recorded in `prompts.json` and surfaced in `00_README.md`, but the per-step text is the same across profiles. Future versions may branch the per-step text on profile.

## Language

`--language` (zh-CN / en) switches the fill-pack content language. Default is `zh-CN` to match the operator's preference. The JSON artifacts (`prompts.json`, `field_checklist.json`, `draft_status.json`) carry a `language` field for downstream tools.

## How an agent should consume the fill pack

A typical loop:

```bash
# 1. Generate draft + fill pack + audit + render.
python3 skills/paper-three-pass-reader/scripts/run_paper_reading.py \
  --input "Attention Is All You Need" \
  --input-kind paper_title \
  --slug myrun \
  --output-root runs/ \
  --fill-pack --audit --render

# 2. Read fill-pack/00_README.md → see what's doable.
# 3. For each step 01..10, read the markdown, fill the corresponding JSON fields.
# 4. After each fill batch, re-run audit + render.
python3 skills/paper-three-pass-reader/scripts/audit_paper_reading.py \
  --input runs/myrun/work/paper_reading.json \
  --json-output runs/myrun/work/audit_result.json

python3 skills/paper-three-pass-reader/scripts/render_page.py \
  --input runs/myrun/work/paper_reading.json \
  --output runs/myrun/paper-reading-output

# 5. Publish.
./skills/paper-three-pass-reader/scripts/publish_output_to_github.sh \
  --output runs/myrun/paper-reading-output \
  --repo conanxin/paper-reading-pages \
  --branch gh-pages \
  --message "Publish myrun"
```

## What the fill pack is NOT

- It is **not** an automatic deep reader. It does not download PDFs, extract figures, or generate claims.
- It does **not** judge whether the paper's content is correct. The audit is structural; correctness is up to the operator.
- It is **not** a SaaS / paper database. No external LLM API, no persistent storage.

## Upgrade path: weak → full

To upgrade `abstract_only` / `screenshot_only` to `full_text`:

1. Place the paper body text in `runs/<root>/<slug>/extracted/full_body.txt` (one paragraph per line, or a markdown file).
2. Re-run the runner with `--reading-mode full_text` (overrides the kind-forced mode).
3. Re-run audit. Claims should now have stronger labels (e.g. `[Paper evidence]` for body-derived claims).
4. Re-run render + publish.

> In v0.2.1-alpha, the runner does not auto-extract the body. The operator must supply the body manually. Future versions may add a stdlib PDF text extractor.

## Failure modes

- **`audit FAIL after fill-pack generation`** — Runner refuses to render (unless `--audit-warn-only`). Check `work/audit_result.json` for the structural issues.
- **`fill-pack/00_README.md` says "draft_fields_count = N" but no drafts in JSON`** — Usually means the runner wrote `[DRAFT]` placeholders that the audit correctly flagged. Replace them.
- **`field_checklist.json` shows everything `unavailable_due_to_reading_mode`** — Reading mode was set to a weak mode; switch to `full_text` and supply body.
