## PROBLEM

By v0.2.9 we had published 9 reading pages to `conanxin/paper-reading-pages`. The first 8 were rendered by older renderer versions (v0.2.5–v0.2.7) and still carry legacy-render artefacts:

- `{% else %}` / `{{ … }}` template-tag leaks.
- Raw Python dict repr (`{'label': …}`) in the Five Cs cards.
- Stale `v0.1.0-alpha` in the page footer.
- Missing Resolver Trail / 解析状态 block (v0.2.8+ feature).
- Missing glossary definitions (v0.2.9+ feature).
- Missing essay-mode markers (v0.2.9+ feature).

We had no automated way to enumerate which live pages have which regressions, so a "should we republish this?" decision had to be made by hand for every page. v0.2.10 adds that automated audit, runnable in one command, with PASS / WARN / FAIL semantics and a recommendation list.

## AUDIT_SCRIPT

`skills/paper-three-pass-reader/scripts/audit_published_pages.py` — stdlib-only (urllib / json / re / argparse). Reads manifest via `--manifest-url` (live) or `--manifest-file` (local). Walks 8 check groups (A–I) and emits JSON + Markdown with per-page status (PASS / WARN / FAIL) and per-issue severity (error / warning / info). Supports `--include-root` (audit the site root too), `--strict` (promote WARN to FAIL), `--warn-only` (always exit 0), `--max-pages` (cap for smoke tests), `--timeout` (per-page HTTP timeout, default 20s), and `--selftest-dir` (offline smoke test).

## RECOMMENDATIONS

Per the audit's `recommendations` block:

- 8 pages still leak template tags → **priority batch re-render + republish** in v0.2.11+.

## LIMITATIONS

- The audit's first live run reports `FAIL` overall because 8 of 9 pages still carry legacy artefacts. This is **expected** and is the audit's intended use; it is not a regression in v0.2.10.
- The audit is read-only. It never writes to `gh-pages`, never triggers a re-render, never modifies any local file outside the output paths passed in.
- The audit only checks what is in the rendered HTML. It cannot check, for example, that a `glossary` term has the right definition content, or that a claim's `evidence_label` is the most appropriate one. Those checks are the audit-pipeline's job, not the audit script's.
- A few of the warnings (`no_visible_claim_id`, `no_evidence_label`, `glossary_no_explicit_definition`) are deliberately `info`-level. They are useful for spotting under-filled pages but should not block a release on their own.
- Some pre-existing v0.2.9 files were re-rendered / fill-pack-walked in this session, which produced small diffs in `runs/you-and-your-research-20260615/you-and-your-research-cn/fill-pack/draft_status.json` and `field_checklist.json`. These are not part of v0.2.10's new work but appear as modified in `git status` and are intentionally excluded from the v0.2.10 commit.
