## PROBLEM

After the v0.2.12-alpha root-index exemption, the live site audit
(`runs/published-pages-audit-20260615-root-index-exemption/audit.json`)
reported a clean PASS for pages and zero error-level / warning-level
issues. One info-level finding remained on the root index:

- `index_no_manifest_link` (severity `info`) — site_index 页面未引用
  published_pages.json (audit 仍可工作,但用户可能无法直接跳转到
  manifest)。

The root index generated by `publish_output_to_github.sh` did not
include any link to the JSON manifest. Users could not click through
to it from the index page, and tools had no machine-readable hint that
the manifest existed at `/published_pages.json`.

## FIX_SUMMARY

Emit the manifest link in two forms from `publish_output_to_github.sh`'s
root index generator, and update `_check_site_index()` in
`audit_published_pages.py` to recognize either form. The live root index
no longer triggers `index_no_manifest_link` after a re-publish.

## PUBLISHER_CHANGE

`skills/paper-three-pass-reader/scripts/publish_output_to_github.sh`
(the inline Python that renders `index.html` from the manifest) now
emits:

1. **`<head>` manifest discovery tag:**

   ```html
   <link rel="alternate" type="application/json"
         href="published_pages.json"
         title="Published pages manifest" />
   ```

2. **Visible About-section link:**

   ```html
   <p>Machine-readable manifest: <a href="published_pages.json">published_pages.json</a>
      &middot; <a href="published_pages.json" hreflang="zh-CN">页面清单 JSON</a></p>
   ```

Both forms use the relative path `published_pages.json` so they work
regardless of where the site is hosted (GitHub Pages, custom domain,
local mirror, etc.).

The publisher regenerates the entire `index.html` on every publish, so
older indexes without the link are replaced naturally on the next
`publish_output_to_github.sh` invocation.

## AUDIT_CHANGE

`_check_site_index()` in `audit_published_pages.py` no longer does a
naive substring check for `published_pages.json`. It now checks for
either:

- A visible `<a href="published_pages.json">` link, OR
- A `<link rel="alternate" type="application/json"
  href="published_pages.json">` discovery tag (matching either
  attribute order).

A root index with neither form still triggers `index_no_manifest_link`
at info level — the check is not silently disabled.

One new selftest fixture was added: `fake-site-index-no-manifest`, a
root index with no link to `published_pages.json`, expected to trigger
`index_no_manifest_link` at info level. The existing `fake-site-index`
and `fake-site-index-leak` fixtures were updated to include the manifest
link so they correctly satisfy the new check.

## LIMITATIONS

- The manifest link is **relative** (`published_pages.json`). It works
  on GitHub Pages, custom domains, and local mirrors because both the
  index and the manifest live at the branch root. If a future deployment
  puts the index at a different URL than the manifest, the relative
  path may break and the audit will start emitting
  `index_no_manifest_link` again.
- The publisher is the **only** source of manifest-link-emitting root
  indexes. Hand-written or third-party indexes will still trigger
  `index_no_manifest_link` until they are regenerated by the publisher
  (or until someone hand-adds the link tag).
- The audit's `index_no_manifest_link` finding remains a single
  info-level advisory. It does not change page status; pages without
  the manifest link still report PASS overall.
